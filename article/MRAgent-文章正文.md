# 复现 MRAgent 之后：轻量实体索引比主动图记忆更实用

> 基于 Hermes Agent 的 A/B 测试

---

## 一、一个场景

你在跟 AI 助手聊天，对话记录有七八轮。你突然想问：

> 「我之前说想做咖啡拉花，后来遇到什么问题来着？」

对话里有三句相关的：
- 第 2 轮：「牛奶打发总是失败」
- 第 4 轮：「换了全脂也不行」
- 第 5 轮：「可能是温度计不准」

这三句分布在 7 轮对话里。你的 AI 助手需要把它们找出来并回答你。

这听起来不难。但实际做起来，不同方案的表现差距很大。

---

## 二、为什么做这个实验

### 起因

我平时用 Hermes Agent[¹](#ref1)，它的 `session_search` 基于 SQLite FTS5 全文检索，大多数场景下够用。但一直有个问题：**当答案分散在多句话里时，单次检索经常漏掉关键信息。**

恰好 ICML 2026 有一篇 MRAgent[²](#ref2)论文提出了解决方案：用图结构记忆加上 LLM 多步遍历，替代传统的一次性检索。

论文在 LoCoMo[³](#ref3)和 LongMemEval[⁴](#ref4)两个 benchmark 上提升很大：

| 方法 | 多跳 F1 | 时间推理 F1 |
|:---|:---:|:---:|
| RAG | 34.89 | 43.52 |
| Mem0 | 45.17 | 58.19 |
| **MRAgent** | **56.72** | **69.82** |

数据很漂亮。但我有个疑问：**让 LLM 在检索阶段做多步推理，速度应该不便宜。**

### 三个问题

我想搞清楚三件事：

1. MRAgent 的改进在我自己的对话场景上能复现吗？
2. 和更轻量的方案比，差距多大？
3. 如果我要改进 Hermes 的检索，最优路径是什么？

于是做了个对比测试。

---

## 三、实验设计

### 三个方案

| 方案 | 做法 | LLM 调用次数 |
|:---|:---|:---:|
| **MRAgent** | 提取实体 → 建 Cue-Tag-Content 图 → LLM 多步遍历 | 3 次 |
| **Flat RAG** | 关键词匹配 → 取 top-5 轮次 → 一次回答 | 2 次 |
| **改进方案** | jieba[⁵](#ref5)建实体索引 → 查索引 → 首轮检索 → 提取新关键词 → 二次扩展 | 1 次 |

### 测试用例

10 个场景，三类：

```
多跳推理（4题）：实体关联、因果关系、否定排除
时间推理（3题）：时序因果、时间区间、精确日期
跨会话（3题）：偏好聚合、事实合并、状态追踪
```

每个场景都有完整对话记录、问题和标准答案。代码和完整数据已开源[⁶](#ref6)。

### 环境

- LLM：Gemma4 12B（Q4_K_M）
- GPU：RTX 4070（远程机器，通过 SSH 隧道调用 Ollama）
- 检索旁路：jieba 分词 + SQLite
- 评测：LLM-as-judge（Gemma4 自评），三级：正确 / 部分正确 / 错误

## 四、结果

### 总表

| 指标 | 改进方案 | MRAgent | Flat RAG |
|:---|:---:|:---:|:---:|
| 全部正确 | **50%** | 20% | 30% |
| 正确+部分正确 | **80%** | 30% | 40% |
| 平均耗时/题 | **11s** | 33s | 4s |
| LLM 调用/题 | **1** | 3 | 2 |
| 稳定运行 | **10/10** | 5/10 | 5/10 |

### 逐题结果

| 题 | 场景 | 改进方案 | MRAgent | Flat RAG |
|:---|:---|---:|:---:|:---:|
| mh1 | Nate 参加什么活动？ | ✓ 7s | ✗ 漏了 | ✓ 7s |
| mh2 | 咖啡拉花的问题？ | ✓ 6s | ✗ 漏了 | ✓ 7s |
| mh3 | 陈教授研究+论文？ | ✗ 截断了 | ✓ **73s** | ✗ 漏了 |
| mh4 | 为啥不选漂流？ | ✓ 9s | ✓ 64s | ✓ 7s |
| tm1 | 为什么停跑步？ | ~ 26s | ~ 47s | ~ 7s |
| tm2 | 发现到维修隔多久？ | ~ 15s | ✗ 挂了 | ✗ 挂了 |
| tm3 | 父母何时到北京？ | ✓ 8s | ✗ 挂了 | ✗ 挂了 |
| cs1 | 跨会话学什么技能？ | ✗ 截断了 | ✗ 挂了 | ✗ 挂了 |
| cs2 | 月租加管理费多少？ | ~ 13s | ✗ 挂了 | ✗ 挂了 |
| cs3 | 谷歌面试结果？ | ✓ 13s | ✗ 挂了 | ✗ 挂了 |

✓=正确  ~=部分正确  ✗=错误或失败

### 几个有意思的点

**1. MRAgent 只赢了一题，但它赢了最难的那题**

mh3 需要连接 3 条信息：张老师退休 → 陈教授接课 → 量子纠错 + Nature Physics。MRAgent 的图遍历确实找到了这个链条，Flat RAG 和我的改进方案都漏了。

**这是 MRAgent 论文的核心论点——主动检索能处理被动检索搞不定的情况。这个论点是成立的。**

代价是 73 秒。

**2. MRAgent 挂了 50% 的题**

超过 6 轮对话的场景，MRAgent 的构建图 prompt 太长，直接返回 HTTP 400。改进方案和 Flat RAG 用同样的模型和接口，没有这个问题。

**3. 简单查询 MRAgent 反而表现最差**

mh1 问「Nate 参加什么活动」——答案在下一句。MRAgent 说「对话里没提到」。因为实体提取阶段出了偏差。**复杂方案在简单问题上反而容易犯错。**

**4. 改进方案的问题是截断策略太粗暴**

我只取了前 4 条相关轮次做证据。mh3 和 cs1 的答案都在第 5-6 轮，被截掉了。这不是方案的问题，是参数没调好。

## 五、改进方案的设计

### 核心思路

MRAgent 的三步：提取线索 → 多步遍历 → 回答

改进方案的三步：jieba 分词 → 查索引 → 两段检索 → 回答

差的那一步——多步遍历——MRAgent 用 LLM 做，我用 SQLite 做。

LLM 做一次 20 秒，SQLite 做一次 2 毫秒。

### 两段检索具体怎么跑

```
第一段：jieba 从问题里切出实体 → 查 SQLite 索引 → 拿到匹配对话轮次
第二段：从第一段结果里提取新实体 → 查还没搜过的轮次
合并 → 送 LLM 一次回答
```

比如问「Nate 参加什么活动」：
1. jieba 切出「Nate」「参加」「活动」
2. 索引里找到含「Nate」的轮次：T0、T1、T2
3. 从 T0-T2 提取新词「电竞比赛」
4. 用「电竞比赛」搜剩余轮次（其实已经够了）
5. LLM 看 T0-T2 的内容，回答：电竞比赛

整个过程 jieba 分词 0.2ms，SQLite 查询 1ms，两步耗时可以忽略。唯一耗时的就是最后一次 LLM 回答，大约 7-8 秒。

### 架构

```
存储：对话 → jieba → 提取实体 → 写入旁路 SQLite
                                       ↓
查询：问题 → jieba → 查索引 → 首轮结果 → 提取新词 → 二次扩展 → LLM 回答
```

SQLite 旁路索引和 Hermes 的 state.db 完全独立，删除旁路索引即可回滚。

### 代价

改进方案贡献的额外延迟：约 2ms（jieba + 两次 SQLite 查询）。对比 MRAgent 额外增加的 40-60 秒（两次额外 LLM 调用）。

## 六、一点思考

### 论文和工程之间有个 gap

MRAgent 证明了「主动检索比被动检索更强」——这是数学上严格证明的定理（定理 4.1：给定相同预算 T，主动策略的表达力严格包含被动策略[²](#ref2)）。

但它没告诉你代价。实际跑下来，用 LLM 做多步遍历的延迟和稳定性代价，工程上很难直接套用。

工程上更实用的路径是：**把检索的过程拆成几步，但每一步都用低成本工具完成，只在最后回答时调用 LLM。**

### 最有价值的思想

论文最有价值的不是那个 Cue-Tag-Content 图结构，而是这句话背后的思路：

> 检索不应该是一次性的，检索结果应该反过来指导后续检索。

这个思路可以用很轻量的方案落地——两段检索就是最简实现。

### 实用的分级策略

```
简单查询 → FTS5 全文检索（毫秒级）
    ↓ 结果不够
多步查询 → 实体索引 + 两段检索（秒级）
    ↓ 还查不到
复杂推理 → LLM 辅助分析（十秒级）
```

90% 的查询在第一步解决，9% 在第二步，1% 需要第三步。这个分级路由可能是最实用的改进。

---

## 七、总结

1. MRAgent 的方向是对的，多步遍历确实能处理单次检索搞不定的复杂问题
2. 但直接用 LLM 做遍历，延迟和稳定性代价太大
3. 用 jieba + SQLite 做同样的事，覆盖了 80% 的场景，代价降低了四个数量级
4. 最有价值的不是方案本身，而是「检索结果指导后续检索」这个思路
